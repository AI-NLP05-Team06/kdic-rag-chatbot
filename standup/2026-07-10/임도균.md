# 분석 필요 페이지 세부 검토

## 1. 예금자보호제도 FAQ

### 1.1 분석 필요 대상으로 분류한 이유

예금자보호제도 FAQ는 실제 사용자의 질문 형태를 포함하고 있기 때문에 일반적으로는 반드시 수집해야 할 데이터처럼 보일 수 있다.

그러나 기존 필수 수집 페이지에 다음 정보가 이미 포함되어 있다.

* 보호한도
* 보호대상 금융회사
* 보호대상 금융상품

FAQ에서도 다음과 같이 유사한 정보를 다룬다.

> 예금보호한도 1억 원은 언제부터 적용되는가?

> 어떤 금융상품이 보호되는가?

> 외화예금도 보호되는가?

따라서 핵심 안내 페이지와 FAQ 사이에 의미적으로 중복되는 데이터가 발생할 수 있다.

### 1.2 우려사항 1: 중복 청크 증가

예를 들어 보호한도 페이지와 FAQ에 다음 내용이 각각 포함될 수 있다.

**보호한도 페이지**

```text
원금과 이자를 합하여 예금자 1인당 1억 원까지 보호한다.
```

**FAQ 페이지**

```text
2025년 9월부터 예금보호한도는 1억 원이다.
```

두 내용을 모두 벡터 데이터베이스에 저장한 상태에서 사용자가 다음과 같이 질문할 수 있다.

> 예금은 얼마까지 보호돼?

이 경우 검색 결과가 다음과 같이 구성될 가능성이 있다.

```text
chunk1: 보호한도 페이지
chunk2: FAQ
chunk3: FAQ
chunk4: FAQ
```

유사한 내용의 청크가 반복해서 검색되면 다음 문제가 발생할 수 있다.

* 검색 결과의 다양성 감소
* 동일하거나 유사한 답변 반복
* 서로 다른 작성 시점으로 인한 최신성 충돌
* 핵심 문서보다 FAQ 청크가 과도하게 검색될 가능성

### 1.3 우려사항 2: 유지보수

FAQ는 정책 변경이나 사용자 문의 사항에 따라 비교적 자주 수정될 수 있다.

예를 들어 시점에 따라 다음처럼 내용이 달라질 수 있다.

**2025년**

```text
예금보호한도 1억 원 시행 예정
```

**2026년**

```text
예금보호한도 1억 원 시행 완료
```

과거 FAQ 데이터가 삭제되지 않고 남아 있으면 최신 핵심 문서와 내용이 충돌할 수 있다. 따라서 FAQ를 수집할 경우에는 페이지의 변경·삭제를 감지하고 기존 데이터를 교체하는 갱신 정책이 필요하다.

### 1.4 결론

예금자보호제도 FAQ가 분석 필요 대상으로 분류된 이유는 다음과 같다.

> 기존 핵심 페이지와 내용이 중복될 가능성이 있으며, FAQ 특성상 변경 주기가 짧아 별도의 최신성 관리가 필요하기 때문이다.

최종적으로는 수집 대상에 포함한다.

FAQ의 자연어 질의응답 데이터는 실제 사용자 질문 대응과 QA 평가셋 구축에 유용하므로, 중복 및 관리 부담보다 활용 가치가 더 크다고 판단하였다.

## 2. 예금자보호한도(해외)

### 2.1 업무 범위 문제

예솜24의 주요 목적은 예금보험공사의 국내 업무를 안내하는 것이다.

사용자는 일반적으로 다음과 같은 질문을 할 가능성이 높다.

> 내가 맡긴 돈은 보호되는가?

> 얼마까지 보호되는가?

> 예금보험금은 어떻게 신청하는가?

그러나 예금자보호한도 해외 페이지는 다음 국가의 예금보호제도와 보호한도를 다룬다.

* 미국
* 일본
* 캐나다
* 영국
* 기타 주요 국가

이 정보는 국내 예금보험 업무 수행을 위한 핵심 데이터라기보다 참고 및 비교 자료의 성격이 강하다.

### 2.2 검색 오염 가능성

사용자가 다음과 같이 질문할 수 있다.

> 예금보호한도는 얼마야?

벡터 검색 결과에서는 다음 두 문서가 동시에 높은 유사도를 가질 수 있다.

```text
문서 A: 한국의 예금보호한도는 1억 원이다.
문서 B: 미국의 예금보호한도는 25만 달러이다.
```

질문에서 국가를 명확하게 지정하지 않으면 해외 자료가 국내 보호한도 질문에 함께 검색될 수 있다.

벡터 검색은 `예금보호한도`라는 의미적 유사성을 중심으로 문서를 검색하기 때문에, 별도의 국가 메타데이터 필터가 없으면 국내 정보와 해외 정보가 혼합될 가능성이 있다.

### 2.3 업데이트 책임

해외 예금보호제도는 국가별 정책 변화에 따라 지속적인 관리가 필요하다.

관리 대상에는 다음 항목이 포함될 수 있다.

* 국가별 예금보호한도 변경
* 통화 및 환율 변동
* GDP 대비 보호 수준 변경
* 제도 운영기관 및 정책 변경
* 비교 기준연도 변경

해외 자료를 수집하면 국내 예금보험 정보뿐만 아니라 여러 국가의 최신 정책까지 지속적으로 확인해야 하는 유지보수 부담이 발생한다.

### 2.4 결론

예금자보호한도 해외 페이지가 분석 필요 대상으로 분류된 이유는 다음과 같다.

> 국내 예금보험 업무 범위를 벗어난 참고성 자료이며, 국내 보호한도 질문에 해외 정보가 혼입될 가능성과 최신성 관리 부담이 있기 때문이다.

따라서 해외 보호한도 데이터는 필수 데이터와 분리하여 선택적으로 수집하는 것이 적절하다.

수집할 경우 다음과 같은 메타데이터를 추가하여 검색 범위를 제한할 필요가 있다.

```yaml
country: 해외
content_scope: 비교자료
collection_priority: optional
```

## 3. 예금보험금 지급대상 금융회사

### 3.1 정보 가치

예금보험금 지급대상 금융회사 정보는 특정 금융회사의 보험금 지급 여부를 확인하는 데 직접적으로 사용될 수 있다.

따라서 정보 가치가 낮아서 분석 필요 대상으로 분류된 것은 아니다. 오히려 실제 사용자 질문 대응 측면에서는 활용 가치가 높은 데이터다.

주요 문제는 해당 페이지의 데이터 제공 방식이다.

### 3.2 정적 웹페이지가 아닌 조회형 데이터

일반적인 안내 페이지는 다음과 같이 고정된 설명문을 포함한다.

```text
예금보험금이란?
→ 예금보험금의 정의와 지급 조건을 설명하는 고정 문서
```

반면 지급대상 금융회사 페이지는 다음과 같은 조회 기능을 제공할 가능성이 있다.

```text
회사명 검색
조건 선택
검색 실행
결과 테이블 출력
```

이 경우 단순한 문서형 웹페이지라기보다 데이터베이스 조회 서비스에 가까운 구조다.

### 3.3 크롤링 난이도

일반적인 정적 웹페이지는 다음 과정으로 수집할 수 있다.

```text
HTTP 요청
  ↓
HTML 응답
  ↓
BeautifulSoup 파싱
  ↓
본문 텍스트 추출
```

하지만 동적 조회 페이지는 다음과 같은 과정으로 동작할 수 있다.

```text
페이지 접속
  ↓
JavaScript 실행
  ↓
검색 조건 입력
  ↓
내부 API 요청
  ↓
결과 테이블 생성
```

따라서 단순한 `requests`와 `BeautifulSoup` 조합으로는 실제 검색 결과를 수집하지 못할 수 있다.

페이지 구조에 따라 다음 방법 중 하나를 선택해야 한다.

* 내부 API 직접 호출
* JavaScript 네트워크 요청 분석
* Playwright 또는 Selenium 기반 동적 크롤링
* 검색 결과를 제공하는 별도 데이터 파일 수집
* 검색 조건별 반복 요청 및 결과 병합

### 3.4 데이터 양과 갱신

지급대상 금융회사 데이터는 시간이 지나면서 변경될 수 있다.

예를 들어 현재 데이터가 다음과 같을 수 있다.

```text
A저축은행
상태: 파산
보험금 지급 상태: 지급 중
```

이후에는 다음과 같은 변화가 발생할 수 있다.

* 신규 지급대상 금융회사 추가
* 보험사고일 변경 또는 추가
* 지급개시일 등록
* 지급 상태 변경
* 연락처 변경
* 기존 항목 삭제 또는 종료 처리

따라서 한 번만 수집하는 것이 아니라 변경 사항을 반영할 수 있는 재수집 구조가 필요하다.

### 3.5 결론

예금보험금 지급대상 금융회사가 분석 필요 대상으로 분류된 이유는 다음과 같다.

> 사용자 질문에 대한 답변 가치는 높지만, 검색형·동적 데이터 특성상 일반 문서와 다른 수집 방식 및 갱신 정책을 결정해야 하기 때문이다.

최종적으로는 수집 대상에 포함하되, 일반 정적 페이지와 분리된 동적 데이터 수집 파이프라인으로 관리하는 것이 적절하다.

본 프로젝트의 목적은 웹페이지를 단순히 크롤링하는 것이 아니라, **RAG 시스템에 활용할 가치가 있는 데이터셋과 갱신 가능한 데이터 파이프라인을 설계하는 것**이다.

* 실제 사용자 질문에 답변하기 위해 필요한 데이터인가
* 기존 핵심 문서와 의미적으로 중복되는가
* 벡터 검색 결과에 노이즈를 발생시킬 가능성이 있는가
* 정책 및 데이터 변경에 따른 최신성 관리가 필요한가
* 정적 페이지인지 동적 조회 데이터인지
* 일반 수집 파이프라인으로 처리할 수 있는가
* 별도의 메타데이터 필터와 갱신 정책이 필요한가

이러한 기준을 바탕으로 수집 대상과 우선순위를 결정하는 것이 단순 크롤링이 아닌, 실제 RAG 데이터 파이프라인 설계자의 관점에 가깝다.
