# 연구 계획서

## AI 서비스 데이터 파이프라인 구축

> 본 프로젝트는 예금보험공사와 금융안심포털에 분산된 웹 데이터를 수집·정제하여, 향후 RAG 기반 예솜24 챗봇이 신뢰도 높은 답변을 생성할 수 있도록 하는 데이터 파이프라인 구축을 목표로 한다.
>
> 우리 팀은 이 과제를 단순한 웹 크롤링 작업이 아니라, **어떤 데이터를 수집해야 하는지 판단하고, 구조를 보존하여 변환하며, 서비스에 필요한 메타데이터와 평가 데이터를 함께 구축하고, 변경사항을 다시 반영할 수 있는 데이터 흐름을 설계하는 문제**로 해석하였다.

---

## 1. Problem Definition

### 1.1 문제점: 비즈니스 니즈

예금보험공사와 금융안심포털에는 예금자보호제도, 예금보험금 안내, 고객 미수령금 신청, 착오송금 반환 신청, 채무조정 안내, 은닉재산 신고 등 여러 업무에 관한 정보가 분산되어 있다.

사용자는 웹사이트의 메뉴와 페이지를 직접 탐색하기보다 다음과 같이 자신의 상황을 중심으로 질문한다.

- “내 예금은 얼마까지 보호돼?”
- “잘못 보낸 돈을 돌려받으려면 어떻게 해야 해?”
- “잘못 받은 사람도 신청할 수 있어?”
- “상속인이 보험금을 신청하려면 어떤 서류가 필요해?”
- “채무조정 신청 절차를 알려줘.”

하지만 현재 웹페이지는 사람이 메뉴를 따라 읽는 구조를 기준으로 설계되어 있어, RAG 챗봇이 바로 검색하고 답변하기에는 다음 문제가 있다.

- 6개 업무의 정보가 여러 페이지와 두 개의 사이트에 분산되어 있다.
- 신청 방법, 대상, 기한, 서류 등 유사한 표현이 여러 업무에서 반복된다.
- 동일하거나 연결된 내용이 여러 페이지에 중복되어 있다.
- 제목, 메뉴, 브레드크럼, 표, 이미지, 첨부파일이 본문과 분리되어 있다.
- 일부 페이지는 정적 문서가 아니라 검색형·동적 데이터에 가깝다.
- 정책이나 페이지 내용이 바뀌어도 기존 적재 데이터가 자동으로 갱신되지 않는다.
- 수집한 데이터가 실제 사용자 질문에 충분히 답할 수 있는지 검증할 평가셋이 부족하다.

따라서 비즈니스 요구사항은 다음과 같이 정의한다.

> 예금보험공사 관련 웹페이지를 AI가 검색·인용할 수 있는 데이터로 변환하고, 6개 업무가 서로 섞이지 않도록 구조화하며, 변경사항을 다시 반영할 수 있는 데이터 파이프라인과 평가용 질의응답 데이터를 구축한다.

### 1.2 Task Reformulation

#### 초기 접근 방법

초기에는 프로젝트를 다음과 같이 이해할 수 있었다.

> 사이트맵에서 URL을 수집하고, 각 페이지의 HTML을 텍스트로 변환하여 저장한다.

그러나 팀 회의에서 업무별 페이지를 직접 조사한 결과, 단순 수집만으로는 해결되지 않는 문제가 확인되었다.

- 채무조정 안내와 착오송금 관련 페이지 사이에 유사한 내용이 존재한다.
- 고객 미수령금 신청의 일부 페이지는 서로 연결되어 있고 내용이 겹친다.
- 채무조정 절차처럼 이미지 안에 핵심 정보가 포함된 페이지가 있다.
- 예금자보호 FAQ와 보호한도 페이지처럼 동일 주제를 다른 형식으로 설명하는 문서가 있다.
- 지급대상 금융회사처럼 정기적으로 바뀌는 조회형 페이지가 있다.
- 불법행위 신고센터처럼 키워드는 유사하지만 은닉재산 신고 업무와는 다른 페이지도 존재한다.
- 어떤 데이터를 챗봇에게 어디까지 제공해야 하는지 먼저 정의할 필요가 있다.

즉, 단순 크롤링은 “페이지를 가져오는 것”에는 성공할 수 있지만 다음 질문에는 답하기 어렵다.

- 어떤 페이지가 실제 답변에 필요한가?
- 내용이 겹치는 페이지를 모두 넣을 것인가?
- 서로 다른 업무의 유사한 문서를 어떻게 구분할 것인가?
- 이미지 안의 절차와 조건은 어떻게 텍스트로 만들 것인가?
- 갱신되는 페이지는 어떤 기준으로 다시 수집할 것인가?
- 데이터가 충분히 적재되었는지는 어떻게 판단할 것인가?

#### 현재 접근 방법

우리 팀은 프로젝트를 크게 두 단계로 재해석하였다.

1. **사이트 → 검색 가능한 데이터로 변환하는 ETL 파이프라인**
   - URL 수집
   - 원본 HTML 및 이미지 확보
   - 구조 보존형 텍스트 변환
   - 변경·삭제 감지
   - DB 또는 적재 형식으로 전달

2. **추출 데이터 → 서비스 품질을 높이는 메타데이터·평가 파이프라인**
   - 업무 및 하위 카테고리 태깅
   - 페이지 요약과 출처 정보 생성
   - 질문 유형과 사용자 시나리오 정의
   - 질의-답변-근거 문서 평가셋 구축
   - 파서나 메타데이터 변경 전후의 결과 비교

멘토링을 통해 두 단계 모두 중요하며, 프로젝트에서 어느 한쪽만 부가 기능으로 보는 것이 아니라 **엔지니어링과 리서치·평가를 5:5 수준으로 함께 다뤄야 한다**고 정리하였다.

### 1.3 Task 설계

본 프로젝트의 최종 Task는 다음과 같이 정의한다.

> **예금보험공사 6개 업무 도메인의 웹 데이터를 결정론적으로 수집·변환하고, 업무별 메타데이터와 평가용 질의응답 데이터를 구성하며, 관리자 트리거를 통해 변경·삭제 내용을 다시 반영할 수 있는 데이터 파이프라인을 구축하는 문제**

세부 Task는 다음과 같다.

#### Task 1. 업무별 수집 범위 정의

- 6개 업무별 사이트와 페이지를 직접 검토한다.
- 페이지가 어떤 질문의 근거가 되는지 기록한다.
- `필수`, `분석 필요`, `선택`, `제외`로 수집 우선순위를 구분한다.
- 유사 키워드를 포함하지만 업무가 다른 페이지를 구분한다.
- 먼저 중요한 질문을 정의하고, 그 질문에 답하기 위해 필요한 문서를 역으로 선정한다.

#### Task 2. 원본 데이터 수집

- 예금보험공사와 금융안심포털의 URL을 수집한다.
- 원본 HTML을 저장한다.
- 이미지와 첨부파일 URL을 함께 수집한다.
- 요청 상태, 수집 시각, 최종 URL을 기록한다.
- 정적 수집이 어려운 페이지는 API 또는 브라우저 자동화 필요 여부를 분류한다.

#### Task 3. 결정론적 HTML → 텍스트 변환

- 원문 변환 과정에는 LLM 사용을 지양한다.
- 동일한 HTML 입력에는 동일한 텍스트가 생성되도록 한다.
- 제목, 소제목, 문단, 목록, 표, 링크, 첨부파일 구조를 유지한다.
- 헤더, 메뉴, 푸터 등 불필요한 영역을 제거한다.
- 파서 수정 전후 결과를 비교할 수 있도록 변환 결과를 버전별로 검증한다.

#### Task 4. 이미지 정보 처리

- 수집 단계에서는 이미지 URL, 대체 텍스트, 캡션, 주변 문맥을 메타데이터로 저장한다.
- 모든 이미지를 일괄적으로 멀티모달 처리하지 않는다.
- 핵심 정보가 이미지에만 존재하는 경우를 먼저 분류한다.
- 이미지 유형별 프롬프트를 구분하여 필요한 이미지만 텍스트화한다.
- 이미지 수가 제한적이면 사람 검수 또는 수작업 입력을 병행한다.

#### Task 5. 메타데이터 스키마 구성

- 6개 업무가 검색 단계에서 섞이지 않도록 `business_function`을 필수 필드로 둔다.
- 사이트 메뉴와 브레드크럼을 기반으로 `sub_category`를 구성한다.
- 청크에서 원본 페이지를 역참조할 수 있도록 `parent_doc_id`를 구성한다.
- 페이지 요약, 출처 URL, 수집일자, 첨부파일, 이미지, 데이터 유형을 기록한다.
- 잘못 보낸 사람과 잘못 받은 사람처럼 같은 업무 안에서도 대상이 다른 경우 사용자 역할 또는 시나리오 태그를 검토한다.

#### Task 6. 청킹 및 적재

- 초기에는 단순 청킹으로 전체 흐름을 먼저 구현한다.
- 이후 소제목, 문단, 표 경계를 고려한 청킹으로 교체 가능하도록 구성한다.
- 페이지와 청크 메타데이터를 JSONL 또는 DB 적재 형식으로 출력한다.
- 최종 파이프라인은 파싱 결과가 실제 검색 가능한 형태로 전달되는 지점까지 포함한다.

#### Task 7. 평가용 질의응답 데이터셋 구축

- 질문을 먼저 만들고 필요한 근거 문서를 확인한다.
- 기존 FAQ와 공식 페이지를 바탕으로 질문·정답·출처를 구성한다.
- 질문 유형과 사용자 페르소나를 정의한다.
- 답변 가능 질문뿐 아니라 답변 불가 질문도 포함한다.
- 파서, 청킹, 메타데이터 변경 후 동일 평가셋으로 결과를 반복 비교한다.
- 프로젝트 종료 후에도 질문을 추가할 수 있는 재사용 가능한 형식으로 만든다.

#### Task 8. 변경 감지 및 갱신

- URL과 변환 본문에 해시 알고리즘을 적용한다.
- 자동 모니터링 전체 구현보다 관리자 트리거 방식의 갱신을 우선한다.
- 트리거 실행 시 신규·변경·삭제 페이지를 판별한다.
- 변경된 문서만 다시 파싱하고 적재한다.
- 파싱 함수 수정 시 전체 또는 선택 영역을 재처리할 수 있도록 한다.

### 1.4 이 문제가 중요한 이유

#### 1. 사용자 질문과 웹사이트 구조는 다르다

웹사이트는 메뉴 중심이지만 사용자는 자신의 상황을 중심으로 질문한다. 따라서 페이지를 그대로 저장하는 것만으로는 실제 질문과 필요한 근거를 연결하기 어렵다.

#### 2. 금융·공공 도메인에서는 문서 선별 자체가 데이터 품질이다

키워드가 유사하더라도 업무가 다른 페이지가 존재한다. 예를 들어 불법행위 신고센터와 은닉재산 신고는 단어상 연관되어 보이지만 실제 업무 범위는 다를 수 있다.

따라서 크롤링 기술만으로는 수집 범위를 결정할 수 없으며, 페이지를 직접 확인하고 도메인 지식을 바탕으로 적재 여부를 판단해야 한다.

#### 3. 6개 업무가 섞이면 답변 오류가 발생할 수 있다

신청 기한, 제출 서류, 대상, 처리 절차와 같은 표현은 여러 업무에서 반복된다. 업무 분류 메타데이터가 없다면 타 업무의 문서가 검색 결과에 혼입될 수 있다.

#### 4. 정적 답변 도메인에서는 평가 데이터의 품질이 중요하다

이번 과제의 답변은 창작형이 아니라 공식 페이지에 근거한 정답형에 가깝다. 따라서 실제 사용 시나리오와 공식 근거를 연결한 평가셋이 있어야 파이프라인이 원하는 답변을 지원하는지 확인할 수 있다.

#### 5. 최신 정보 반영은 서비스 신뢰성과 연결된다

예금보호한도, 신청 조건, 지급대상 금융회사 등은 변경될 수 있다. 과거 문서와 최신 문서가 함께 남으면 정확한 답변을 보장하기 어렵다.

#### 6. 파이프라인은 일회용 수집 코드가 아니어야 한다

이번에 한 번 데이터를 모으는 것으로 끝나는 것이 아니라, URL이 추가되거나 페이지가 변경되고 파서가 수정되어도 다시 사용할 수 있어야 한다. 따라서 수집·파싱·메타데이터·평가·갱신 단계가 분리된 구조가 필요하다.

## 2. Background & Baseline

### 2.1 Related Approaches

#### 단순 정적 크롤링

`requests`와 `BeautifulSoup`을 이용해 HTML을 수집하고 텍스트를 추출한다.

- 장점: 구현이 간단하고 빠르다.
- 한계: 동적 데이터, 표, 이미지, 첨부파일, 메뉴 구조 보존이 어렵다.

#### 브라우저 자동화 기반 크롤링

Selenium 또는 Playwright로 JavaScript 실행 이후의 페이지를 수집한다.

- 장점: 동적 페이지와 조회 결과 수집에 유리하다.
- 한계: 실행 비용이 높고 모든 페이지에 적용하면 불필요하게 복잡해진다.

#### 범용 본문 추출기

Trafilatura, Readability, Unstructured 등을 이용해 본문을 자동 추출한다.

- 장점: 다양한 페이지에 빠르게 적용할 수 있다.
- 한계: 공공기관 사이트의 표, 첨부파일, 이미지 문맥 보존 여부를 별도로 검증해야 한다.

#### LLM 기반 변환

HTML이나 이미지를 LLM 또는 멀티모달 모델로 텍스트화한다.

- 장점: 복잡한 이미지와 비정형 구조를 설명할 수 있다.
- 한계: 결과가 비결정적이며 누락·재해석 가능성이 있다.

우리 팀은 원문 HTML 변환에는 결정론적 파서를 사용하고, LLM은 페이지 요약이나 선별된 이미지 설명 등 보조 메타데이터 생성에 제한적으로 사용한다.

### 2.2 이번 프로젝트의 Baseline

#### Baseline Pipeline

```text
업무별 URL 목록
        ↓
requests 기반 HTML 수집
        ↓
원본 HTML 저장
        ↓
BeautifulSoup 기반 본문 추출
        ↓
제목·문단·목록·표 변환
        ↓
글자 수 기반 단순 청킹
        ↓
기본 메타데이터 연결
        ↓
JSONL 저장
```

#### Baseline 메타데이터

```yaml
doc_id: 원본 페이지 식별자
chunk_id: 청크 식별자
source_url: 출처 URL
site_name: 예금보험공사 또는 금융안심포털
title: 페이지 제목
business_function: 6개 업무 중 소속 업무
sub_category: 업무 내 하위 분류
breadcrumb: 메뉴 및 페이지 경로
collection_priority: 필수/분석필요/선택/제외
content_type: 정적 페이지/FAQ/동적 조회/이미지 중심
collected_at: 수집 시각
content_hash: 변경 확인용 해시
parent_doc_id: 원본 문서 ID
attachments: 첨부파일 목록
images: 이미지 목록
```

#### Baseline 선정 이유

- 먼저 전체 ETL 흐름이 동작하는지 빠르게 확인할 수 있다.
- 원본 HTML과 변환 결과를 직접 비교할 수 있다.
- 파싱 오류와 메타데이터 오류를 단계별로 분리할 수 있다.
- 이후 파서, 청킹, 이미지 처리 방식을 변경했을 때 비교군으로 사용할 수 있다.

## 3. Proposed Method

### 3.1 핵심 아이디어

우리 팀의 핵심 접근은 다음과 같다.

> **중요한 사용자 질문을 먼저 정의하고, 그 질문에 필요한 공식 문서를 선별한 뒤, 웹페이지 구조를 보존한 결정론적 ETL 파이프라인과 서비스용 메타데이터·평가 파이프라인을 함께 구축한다.**

이를 위해 다음 원칙을 적용한다.

1. 많이 수집하는 것보다 필요한 데이터를 선별한다.
2. 원문 변환은 결정론적으로 수행한다.
3. 업무·사용자 역할·문서 유형을 메타데이터로 구분한다.
4. 이미지와 동적 페이지는 유형을 확인한 후 선택적으로 처리한다.
5. 해시와 관리자 트리거를 이용해 변경사항을 다시 반영한다.
6. 평가셋을 일회성 결과물이 아니라 지속적으로 확장 가능한 데이터로 만든다.
7. 파서나 스키마 변경의 효과를 동일 평가셋으로 비교한다.

---

### 3.2 Baseline 대비 개선점

| 구분 | Baseline | Proposed |
| --- | --- | --- |
| 수집 범위 | 조사한 URL을 그대로 수집 | 질문-근거 문서 관계와 도메인 검토를 거쳐 선별 |
| 업무 구분 | 업무명 단일 태그 | 업무·하위분류·사용자 역할·문서 유형 구분 |
| 본문 추출 | 일반 본문 파싱 | 제목·표·링크·첨부파일·이미지 문맥 보존 |
| 이미지 | URL만 저장 | 중요 이미지 선별 후 유형별 텍스트화 및 사람 검수 |
| 동적 데이터 | 수집 실패 가능 | API/브라우저 자동화 필요 여부를 별도 분류 |
| 중복 | 동일 URL 중심 확인 | 연결 페이지·FAQ·내용 중복까지 검토 |
| 변경 관리 | 전체 재수집 | 해시 기반 변경 판별 + 관리자 트리거 |
| 평가 | 수집 성공 여부 | 질문·정답·근거·사람 평가를 연결한 시나리오 평가 |
| 개선 검증 | 결과 파일 비교 | 파서·청킹·메타데이터 변경 전후 검색 결과 비교 |

### 3.3 Dataset & Preprocessing

#### 데이터 소스

- 예금보험공사: `https://www.kdic.or.kr`
- 금융안심포털: `https://fins.kdic.or.kr`

두 사이트 중 하나를 주 데이터 소스로 고정하지 않는다. 업무별로 사용자 질문에 더 적절하고 최신성이 높은 공식 페이지를 선택하며, 필요한 경우 두 사이트의 문서를 함께 수집한다.

#### 대상 업무

1. 예금자보호제도
2. 예금보험금 안내
3. 고객 미수령금 신청
4. 착오송금 반환 신청
5. 채무조정 안내
6. 은닉재산 신고

#### 초기 사이트 분석 결과

| 업무 | 필수 | 분석 필요 | 합계 |
| --- | ---: | ---: | ---: |
| 예금자보호제도 | 5 | 2 | 7 |
| 예금보험금 안내 | 3 | 1 | 4 |
| 고객 미수령금 신청 | 3 | 1 | 4 |
| 착오송금 반환 신청 | 17 | 5 | 22 |
| 채무조정 안내 | 3 | 5 | 8 |
| 은닉재산 신고 | 2 | 2 | 4 |
| **합계** | **33** | **16** | **49** |

> 위 표는 프로젝트 착수 전 초기 분석이다. 실제 수집과 평가 과정에서 질문에 필요한 문서가 추가되거나, 중복·범위 부적합 문서가 제외될 수 있다.

#### 페이지 판단 기준

- 실제 사용자 질문의 근거가 되는가
- 업무 범위와 직접 관련되는가
- 동일 내용을 설명하는 다른 페이지가 있는가
- 최신 정보 또는 예외 조건을 포함하는가
- 이미지·표·첨부파일에 핵심 내용이 있는가
- 정적 문서인가, 동적 조회 데이터인가
- 사용자 역할에 따라 답변이 달라지는가
- 변경 가능성과 갱신 필요성이 높은가
- 평가 질문 생성에 활용할 수 있는가

#### 전처리 흐름

```text
URL 및 업무 범위 정의
        ↓
원본 HTML·이미지·첨부파일 정보 수집
        ↓
본문 영역 및 문서 유형 판별
        ↓
제목·소제목·문단·목록·표 변환
        ↓
링크·첨부파일·이미지 주변 문맥 보존
        ↓
중복·연결 페이지 후보 기록
        ↓
페이지 메타데이터 생성
        ↓
청킹 및 parent-child 연결
        ↓
JSONL 또는 DB 적재 형식 출력
```

#### 원본 보존 구조

```text
data/
├── raw/
│   ├── html/
│   └── assets/
├── parsed/
│   ├── markdown/
│   └── image_text/
├── processed/
│   ├── documents.jsonl
│   └── chunks.jsonl
└── evaluation/
    └── qa_dataset.jsonl
```

### 3.4 이미지 처리 설계

이미지는 일괄 멀티모달 처리하지 않고 다음 유형으로 구분한다.

| 이미지 유형 | 예시 | 처리 |
| --- | --- | --- |
| 장식 이미지 | 배너, 아이콘 | URL만 기록하거나 제외 |
| 본문 보조 이미지 | 본문 내용을 반복하는 도식 | 주변 문맥과 함께 저장 |
| 핵심 절차 이미지 | 채무조정 절차도 | 멀티모달 텍스트화 또는 사람 검수 |
| 수치·표 이미지 | 비교표, 통계 그래프 | 구조화 텍스트 변환 필요 여부 검토 |
| 링크형 이미지 | 신청 버튼, 외부 페이지 연결 | 링크 목적과 URL 기록 |

이미지 텍스트화가 필요한 경우 이미지 유형별 프롬프트를 분리하여 결과 형식을 통일한다.

```yaml
image_type: process_diagram
description: 단계별 절차 설명
source_url: 원본 페이지 URL
image_url: 이미지 URL
verified_by_human: true
```

### 3.6 갱신 파이프라인 설계

멘토링 결과, 초기 프로젝트에서는 실시간 모니터링보다 **관리자 트리거 방식**이 구현 가능성과 검증 측면에서 적절하다고 판단하였다.

#### 변경 판별

```text
기존 URL 목록 + 신규 URL 목록
        ↓
new / existing / deleted 판별
        ↓
existing 페이지의 content_hash 비교
        ↓
unchanged / updated 판별
```

#### 상태와 처리

| 상태 | 의미 | 처리 |
| --- | --- | --- |
| `new` | 신규 페이지 | 수집·파싱·메타데이터 생성·적재 |
| `unchanged` | 변화 없음 | 재처리 생략 |
| `updated` | 본문 또는 구조 변경 | 기존 문서·청크 교체 |
| `deleted` | 페이지 삭제 | 비활성화 또는 적재 데이터 제거 |
| `failed` | 수집 또는 파싱 실패 | 오류 기록 및 재시도 |

#### 실행 예시

```bash
python run_pipeline.py --mode full
python run_pipeline.py --mode update
python run_pipeline.py --business-function "예금자보호제도"
```

파싱 함수가 변경된 경우 원본 HTML을 다시 다운로드하지 않고 저장된 원본을 이용해 재파싱할 수 있도록 단계별 실행을 분리한다.

```text

    ↓

    ↓

    ↓

    ↓

```

평가셋은 프로젝트 종료 후에도 새 질문을 추가하고 다시 실행할 수 있는 형태로 구축한다.

### 3.8 검증 가설

#### H1. 업무 메타데이터는 도메인 간 문서 혼입을 줄일 것이다

전체 검색과 업무 필터 검색을 비교했을 때, 업무 필터를 적용한 결과에서 타 업무 근거 문서 비율이 낮아질 것으로 예상한다.

#### H2. 구조 보존형 파싱은 답변에 필요한 조건과 절차의 누락을 줄일 것이다

단순 평문 변환보다 표·소제목·링크 구조를 유지한 문서가 정답 근거를 더 완전하게 보존할 것으로 예상한다.

#### H3. 질문 중심의 수집 범위 정의는 불필요한 데이터 적재를 줄일 것이다

사이트 전체를 일괄 수집하는 방식보다 질문-근거 관계를 먼저 정의하면 범위 밖 문서와 유사 키워드 문서의 혼입을 줄일 수 있을 것이다.

#### H4. 해시 기반 관리자 트리거는 갱신 비용을 줄일 것이다

변경되지 않은 페이지의 재처리를 생략하면서도 변경·삭제 페이지를 반영할 수 있을 것이다.

#### H5. 시나리오 기반 평가셋은 단순 검색지표보다 서비스 품질을 잘 설명할 것이다

문서 검색 여부뿐 아니라 사용자 조건, 답변 가능 범위, 공식 근거와의 일치도를 함께 평가할 수 있을 것이다.

## 4. Experiment Design

### 4.1 수행할 실험

#### E1. 업무별 수집 범위 검증

- 필수·분석 필요·선택·제외 페이지 재검토
- 유사 키워드이지만 업무 범위가 다른 페이지 확인
- 질문별 근거 문서가 최소 1개 이상 존재하는지 확인
- 중복·연결 페이지의 역할 구분

#### E2. HTML → 텍스트 변환기 비교

- BeautifulSoup 직접 파싱
- 범용 본문 추출기
- 사이트 구조 기반 커스텀 파서

평가 항목:

- 제목·소제목 보존
- 문단 순서
- 표 구조
- 링크·첨부파일
- 이미지 정보
- 불필요 영역 제거
- 동일 입력에 대한 동일 출력

#### E3. 이미지 처리 범위 검증

- 전체 이미지 수와 유형 파악
- 이미지 없이 답변 가능한 페이지 비율 확인
- 핵심 정보 이미지 선별
- 멀티모달 텍스트화 결과와 사람 작성 결과 비교

#### E4. 메타데이터 스키마 검증

- `business_function`만 사용한 경우
- `business_function + sub_category`를 사용한 경우
- 사용자 역할까지 구분한 경우

질문과 문서의 분류 일치 여부를 확인한다.

#### E5. 청킹 비교

- 글자 수 기반
- 문단 기반
- 소제목·표 기반

정답 근거가 하나의 청크에 보존되는지 비교한다.

#### E6. 갱신 파이프라인 검증

저장된 테스트 HTML을 이용해 다음 상황을 재현한다.

- 페이지 추가
- 본문 수정
- 첨부파일 변경
- 이미지 변경
- 페이지 삭제
- 파싱 함수 변경 후 재처리

#### E7. 평가셋 시나리오 검증

- 질문 유형별 분포 확인
- 페르소나별 질문의 현실성 검토
- 답변 가능·불가 질문 구분
- 공식 근거와 정답 일치 여부 사람 검수

## 9. 프로젝트 해석 및 최종 방향

우리 팀은 본 프로젝트를 다음과 같이 해석하였다.

> 이 프로젝트의 핵심은 웹페이지를 많이 수집하는 것이 아니라, **고객이 실제로 묻는 질문에 필요한 공식 데이터를 선별하고, 6개 업무가 섞이지 않도록 구조화하며, 데이터가 변경되어도 다시 반영할 수 있는 ETL·메타데이터·평가 파이프라인을 구축하는 것**이다.

팀 회의와 멘토링을 통해 다음 방향을 확정하였다.

1. 두 공식 사이트 중 하나를 주 데이터 소스로 고정하지 않는다.
2. 중요한 질문을 먼저 정의하고 필요한 페이지를 수집한다.
3. 금융 도메인을 이해한 뒤 페이지별 적재 가치를 판단한다.
4. 원문 텍스트 변환에는 LLM을 사용하지 않는다.
5. 이미지 처리는 전체 자동화보다 필요한 데이터의 규모와 유형을 먼저 확인한다.
6. 6개 업무와 사용자 역할이 섞이지 않도록 메타데이터를 설계한다.
7. 해시와 관리자 트리거 방식으로 변경·삭제를 반영한다.
8. 파싱 결과는 실제 검색 또는 DB 적재 가능한 형태까지 전달한다.
9. 평가셋은 시나리오·공식 정답·근거·사람 평가를 연결한다.
10. 파서와 메타데이터를 개선할 때 동일한 평가셋으로 결과 변화를 비교한다.
11. 엔지니어링과 평가·도메인 분석을 동일하게 중요한 영역으로 다룬다.
12. 프로젝트 진행 중의 결정과 실패 사례도 GitHub에 지속적으로 문서화한다.
